# Previous Environment Commit Status Controller

The Previous Environment Commit Status controller gates each environment on the **previous
environment** in a linear pipeline: an environment may only promote a dry commit once the
environment before it has promoted the same commit and is healthy. This is the common
dev → staging → prod case.

A linear pipeline is just a chain-shaped dependency graph, so this controller is a thin adapter over
the [DAGCommitStatus](dag-commit-status.md) controller: it takes the PromotionStrategy's environments
in spec order, builds a chain where each environment `dependsOn` the one before it, and hands that to
the DAG controller, which does the actual gating. Use this controller when your pipeline is linear;
use DAGCommitStatus directly when you need fan-out/fan-in.

> [!IMPORTANT]
> The gate is not created or injected automatically. You must create a
> PreviousEnvironmentCommitStatus for each PromotionStrategy you want to gate, and add its `key` to
> that PromotionStrategy's global `proposedCommitStatuses`. See [Wiring the gate into the
> PromotionStrategy](#wiring-the-gate-into-the-promotionstrategy) below.

## Example Configuration

```yaml
apiVersion: promoter.argoproj.io/v1alpha1
kind: PreviousEnvironmentCommitStatus
metadata:
  name: demo
spec:
  key: promoter-previous-environment
  promotionStrategyRef:
    name: demo
```

The controller derives the chain from the referenced PromotionStrategy's `environments` (in spec
order), so there is no graph to declare here — that is the difference from DAGCommitStatus.

### `spec.key`

`spec.key` is the gate name your PromotionStrategy checks in `proposedCommitStatuses`. It is
required and must match a key declared in that PromotionStrategy's `proposedCommitStatuses`, so the
gate this controller produces is actually enforced. A common value is
`promoter-previous-environment`.

## Wiring the gate into the PromotionStrategy

The PreviousEnvironmentCommitStatus only *produces* the gate (via the DAGCommitStatus it generates);
the PromotionStrategy must *consume* it. Add the same `key` to the PromotionStrategy's global
`proposedCommitStatuses`:

```yaml
apiVersion: promoter.argoproj.io/v1alpha1
kind: PromotionStrategy
metadata:
  name: demo
spec:
  proposedCommitStatuses:
    - key: promoter-previous-environment  # same as PreviousEnvironmentCommitStatus.spec.key
  environments:
    - branch: environments/development
    - branch: environments/staging
    - branch: environments/production
  gitRepositoryRef:
    name: demo
```

> [!IMPORTANT]
> As a safety check, the PromotionStrategy controller fails its reconcile if a DAGCommitStatus
> (including one generated by a PreviousEnvironmentCommitStatus) references the PromotionStrategy but
> its `key` is not present in the PromotionStrategy's `proposedCommitStatuses` — otherwise the gate
> it produces would never be enforced. This safety check is intended to be removed in v1.0; see
> [Roadmap](../../roadmap.md).
